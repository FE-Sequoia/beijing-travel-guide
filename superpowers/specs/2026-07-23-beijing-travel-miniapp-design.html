北京旅游攻略微信小程序设计方案 ​

1. 目标与边界 ​

将现有 VitePress 项目 beijing-travel-guide 的攻略内容逐步转化为一款供个人使用的原生微信小程序。小程序不依赖后端、不需要正式发布，所有攻略内容由项目内的 JSON 静态数据驱动。

第一版的目标是完成可用且视觉完整的框架，并迁移少量代表性页面来验证浏览、检索和收藏流程；其余内容在后续迭代中逐步补齐。

明确不做：账号体系、云数据库、服务端接口、支付、正式发布流程、在线地图和实时数据。

2. 技术方案 ​

  • 使用原生微信小程序 JavaScript、WXML、WXSS 和 JSON 配置文件。
  • 不引入 UniApp、TypeScript、第三方组件库或构建步骤。
  • 内容以本地 JSON 文件保存；页面通过统一的数据访问模块读取，不直接依赖数据文件的内部结构。
  • 收藏和阅读记录使用微信本地存储，仅在当前设备和当前微信环境中保存。

3. 信息架构与 TabBar ​

底部采用四个稳定入口：

Tab目的首屏内容
首页快速开始与推荐搜索入口、分类入口、趣玩推荐、精选内容、近期阅读
探索浏览全部景点五大分类、关键词搜索、筛选和景点列表
攻略行前与行程信息最佳时间、交通、门票、路线、美食、住宿、注意事项
我的离线个人状态收藏、阅读历史、数据说明和简单设置

分类列表、搜索结果和景点详情均为非 Tab 页面，从首页或探索页进入。Tab 图标使用简洁线性风格：首页为屋檐、探索为罗盘、攻略为册页、我的为人物；选中态使用青绿色，未选中态使用浅灰蓝。

4. 页面与功能 ​

首页 ​

首页包含搜索入口、五大分类入口、趣玩模块、精选内容和近期阅读。趣玩模块使用横向滑动的小卡片，每张卡片展示封面、名称、分类和一句推荐文案。首批固定推荐内容为：

  1. 天安门广场
  2. 中国国家博物馆
  3. 故宫博物院
  4. 曹雪芹故居

趣玩内容由 JSON 中的推荐标记与排序控制，调整或新增推荐项无需修改页面代码。

探索 ​

展示名胜古迹、近代历史、宗教建筑、博物馆和城市公园五大分类。用户可按分类浏览、使用本地关键词搜索，并进入景点详情。

景点详情 ​

展示名称、分类、封面、摘要、标签、实用信息、分段正文和相关景点。正文由结构化内容区块渲染，避免在运行时解析 Markdown。

攻略 ​

展示最佳旅行时间、交通、门票预约、推荐路线、美食、住宿和注意事项等实用主题。

我的 ​

展示本地收藏和阅读历史。数据仅保存在本地；若卸载小程序或清除微信缓存，记录会丢失。

5. 数据设计与迁移 ​

建立 data/ 目录，按景点、分类、指南和路线拆分 JSON。景点的统一字段至少包括：id、名称、分类、封面、摘要、标签、开放/预约信息、正文区块、关联景点与推荐配置。

迁移分两步：

  1. 首版先录入趣玩中的四个景点，并在各分类中补充少量代表景点，验证完整的浏览和搜索流程。
  2. 后续将现有 VitePress Markdown 按目录逐步整理为结构化 JSON，完成全量内容迁移。

现有 SVG 资源优先复用。尚无真实封面的内容使用统一的渐变/插画占位样式；后续仅需替换 JSON 中的资源路径。

6. 视觉方向 ​

采用清新、克制的“北京晨光”风格：暖米白作为页面底色，青绿色作为主色，朱砂红仅用于重点强调。页面保持充足留白，使用轻量圆角卡片、柔和阴影和层级明确的排版，避免厚重的仿古视觉。

7. 异常状态与验证 ​

  • JSON 缺少内容时显示友好的空状态。
  • 图片加载失败时显示统一渐变占位。
  • 搜索无结果时显示说明和清空筛选的操作入口。
  • 本地存储失败不阻断阅读，仅提示收藏或历史记录未保存。

每阶段均在微信开发者工具中导入、编译并验证：四个 Tab 的切换、页面跳转、JSON 渲染、本地搜索、收藏、阅读历史和异常空状态。

8. 实施阶段 ​

  1. 搭建原生 JavaScript 小程序骨架、四个 TabBar 页面与统一设计基础。
  2. 完成首页、趣玩模块和四个首批静态景点。
  3. 完成探索页、分类列表、详情页、本地搜索、收藏和阅读历史。
  4. 完成攻略页，并按现有 VitePress 目录逐步迁移剩余内容。
