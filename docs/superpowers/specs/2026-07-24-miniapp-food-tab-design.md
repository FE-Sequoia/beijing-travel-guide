# 北京慢游记：美食 Tab 设计说明

## 目标

为原生 JavaScript 微信小程序新增一个可离线浏览的“美食”Tab。首版以 12 家北京代表性餐厅／品牌构成精选清单，提供清晰的双列图文浏览与可读的餐厅详情，不依赖后端或云数据库。

## 范围与约束

- 使用现有原生微信小程序技术栈与本地 CommonJS 静态数据模块；不添加 npm 依赖。
- 内容数据放在 `beijing-travel-mp/data/foods.js`，以可读的本地对象维护。
- 资料基于美团、大众点评、抖音及品牌公开页面交叉整理为原创摘要；不复制平台评论、榜单或长篇文案。
- 所有封面使用 HTTPS 网络实拍图，并为图片加载失败提供页面级兜底。
- 首版固定收录 12 家，覆盖烤鸭、铜锅涮肉、京味小吃、面食、宫廷菜与老北京甜品；不做搜索、排序、点评、下单或地图导航。
- 推荐指数以整数 `rating`（1–5）保存并在界面渲染为实心／空心星；首版均为精选餐厅，评分不等同于平台评分。
- 继续使用现有本地收藏模块；美食与景点共享收藏 ID 命名空间，食物 ID 使用 `food-` 前缀避免冲突。

## 信息架构

`app.json` 新增 `pages/food/index` 作为第三个 Tab，将现有“攻略”和“我的”依次后移；新增非 Tab 详情页 `pages/food-detail/index`。页面注册顺序保证全部 Tab 页排在普通详情页之前。

### 美食列表

顶部采用既有编辑感页头，标题为“吃在北京”，副标题说明为本地精选。主体为两列等宽卡片栅格：上部封面图，下部餐厅名和 1–5 颗星。卡片高度根据图片比例统一，左右和列间距采用全局设计 token 的节奏，不引入横向溢出或瀑布流不齐边问题。点击卡片以 `wx.navigateTo` 打开详情。

### 美食详情

详情页依次呈现：网络实拍图、餐厅名称与收藏按钮、推荐指数、简介、代表菜／食物发展历史、品牌历史、就餐地址、就餐提示。四个长文区域由可复用的本地数组字段驱动，每段保留标题和短正文，避免内容堆叠成一整块。地址使用文本呈现，不调用地图或外部服务。

## 数据模型

每一项为：

```js
{
  id: 'food-quanjude-qianmen',
  name: '全聚德（前门店）',
  cover: 'https://…',
  rating: 5,
  summary: '一句原创简介',
  signature: '推荐菜或点单方向',
  address: '北京市…',
  tips: '排队、预约或点单提示',
  sections: [
    { title: '美食发展历史', body: '…' },
    { title: '品牌历史', body: '…' }
  ]
}
```

数据访问层新增 `getFoods()`（按名称稳定排序或按手工精选顺序返回副本）及 `getFoodById(id)`，使页面不直接 `require` 数据文件。详情页接收 `id`，找不到数据时复用现有空态组件并提供返回操作。

## 视觉与交互

沿用现有米白背景、墨绿和朱砂强调色。卡片使用轻薄边框、圆角与阴影，不再在卡片内显示冗长标签。星级采用纯文本星形或 WXML 循环，避免依赖图标字体和 Unicode 兼容问题。图片失败时保持同尺寸的温和色块与餐厅名称，避免布局跳动。所有新页面根节点为普通 `view`，不使用根级垂直 `scroll-view`，以保持既有兼容性修复。

## 验收与测试

- `app.json` 有且仅有五个 Tab，第三项指向 `pages/food/index` 并标注“美食”。
- `foods.js` 含 12 条带 `food-` 前缀的唯一记录；每条都有 HTTPS 封面、1–5 整数评分、名称、简介、招牌、地址、提示和两段历史内容。
- `utils/data.js` 的美食访问函数正确返回列表、详情和空结果。
- 列表与详情 WXML、JS、WXSS 均存在，卡片点击链接到详情；详情可收藏、可处理缺失 ID、可处理封面加载失败。
- 项目结构测试扩展为覆盖 Tab、路径、数据和禁止新页面根级垂直 `scroll-view`。
- 运行现有数据、存储、项目测试、全部 JavaScript 语法检查和 `git diff --check`。

## 非目标

不接入美团、大众点评、抖音的 API 或爬虫；不显示第三方评分、用户评论、实时营业信息、订位、点单、支付、地图导航、后台管理和云开发。
